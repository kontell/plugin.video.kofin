# plugin.video.kofin — Phase 4 implementation plan

Date: 2026-07-18. Operationalizes build phase 4 from `rewrite-research.md` §11 and feature §8.1 — the **SyncPlay port**. The design report is `ref/syncplay-report.md` §9 (the jellyfin-kodi SyncPlay implementation plan); it governs the flows and Kodi-capability decisions. The fork (`ref/jellyfin-kodi`, branch `combined/syncplay-sync`, `jellyfin_kodi/syncplay/`) is the source — a ~2.3k-line package (manager 1129 + playback 757 + timesync 119 + ui 106 + utils 197) with ~1.8k lines of tests (`tests/test_syncplay*.py`) — that ports as a package under the same transplant discipline phase 2 used for the sync engine. Test gates reference `testing-plan.md` (§S4), which this plan expands from its current single-paragraph form into S4.1–S4.7 (a doc update landed in the same commit, as phases 2/3 did). Phases 1–3 are complete, merged to `main`, and live-verified. **This plan deviates from report §4 in one place** — SyncPlay settings go into the existing **Playback** tab as a third group, not a dedicated tab (§4 below) — recorded here the way phase 3 recorded its report §8.2 deviation.

**Phase 4 deliverable**: a "SyncPlay" root entry (gated on `syncPlayEnabled`) that lists/creates/joins/leaves groups; once in a group, Kodi follows the group's timeline — scheduled Unpause/Pause/Seek/Stop execute against an offset-corrected clock (NTP-style timesync over `/GetUtcTime`), Ready/Buffering are reported, and a drift controller holds position with `Player.SetTempo` (micro-seek fallback where tempo is unavailable); user pause/seek/next in Kodi drive the group; and the phase-3 `syncplay_group_active` flag is finally driven — inside a group, Play Next (and cinema/near-end prompts) are withheld because the group queue is authoritative. Everything is service/plugin-side — no writer changes, no schema touch, no server-plugin work. Protocol **v1**-compatible (the report's v2 is server-side, adopted opportunistically — §7).

---

## 1. Scope

**In**: the `syncplay/` package transplant (`manager.py` group membership + WS-message mirror + dispatcher thread, `playback.py` command scheduling + drift controller, `timesync.py` NTP-over-`/GetUtcTime`, `ui.py` the group menu, `utils.py`); the ~18 `/SyncPlay/*` + `/GetUtcTime` typed additions to `core/api.py`; a SyncPlay **root entry** and its `mode=syncplay` → IPC `SyncPlayMenu` → service-side menu on a worker thread; `onAVStarted` added to `service/player.py` plus guarded forwarding of the playback callbacks into the manager; wiring the WS `SyncPlayCommand`/`SyncPlayGroupUpdate` messages into the manager; the manager driving `player.syncplay_group_active` (the phase-3 stub); the SyncPlay settings **group inside the Playback tab**; the S4.x live gates against `ref/syncplay-conformance` as the scripted second member.

**Out (later phases / deliberately)**: KofinSyncQueue and tier-1 (phase 5); plugin-free fallback (phase 6); Piers live testing (the syncplay code is schema-agnostic — no DB writes — so L2 is N/A; Piers is a runtime smoke in hardening); **protocol v2** client work (kofin stays v1-compatible; v2 robustness is server-side, report §7 — adopt by capability probe when it lands, never raise a floor); a full group-**queue-editing** UI (kofin *mirrors* the group queue so next/prev and tail edits work — report §9.5.1 — but does not build add-to-group-queue browsing in v1); external-player SyncPlay (unsupported — the entry is hidden when a non-kofin external player is selected, report §9.5.5); theme/cinema *inside* a group (suppressed, not ported into the group timeline).

## 2. Decisions locked in phase 4

* **Transplant discipline, again.** The `syncplay/` package ports with *mechanical* adaptation only — imports (`..helper` → kofin `core`/`plugin`), `LazyLogger` → `Logger`, `translate()` fork ids → kofin string ids, `settings()`/`window()`/`dialog()` shims → kofin `settings`/`state`/`xbmcgui.Dialog`, `Jellyfin()` client → the kofin `Api`. The scheduling math, the drift controller, the timesync min-RTT window, the guard-flag echo suppression, and the dispatcher-thread ordering stay recognizably identical — they are the proven parts (the report's whole point is that Kodi is a *favourable* SyncPlay platform and this code already implements §9's flows). `mypy` runs **relaxed over `kofin/syncplay/**`** (per-module `disallow_untyped_defs = False`, exactly as phase 2 did for `kofin/sync/**`), tightening opportunistically. The fork's `tests/test_syncplay*.py` port with the package and are the correctness proof — no rewrite pass.
* **The one substantive adaptation: playback start re-targets to kofin's plugin play path.** The fork's `playback.py` opens a *resolved* MediaSource path directly (`actions.get_playlist(item)` → `playlist.add(item["PlaybackInfo"]["Path"], …)`, `playback.py:321-329`) because the fork owns an `Actions` playlist builder. kofin has no such thing — playback flows `plugin://plugin.video.kofin/?mode=play&id=<id>` → `plugin/play.py` resolves (device profile, direct-play/transcode decision, native resume) → pushes to `kofin.play.json` → the service `Player` reports. So the SyncPlay port builds its `xbmc.PlayList(VIDEO)` from **kofin plugin URLs** (exactly as `service/remote.py:66` already does for remote Play), not from resolved paths. Consequence: device-profile selection, the transcode ladder, resume, and playback reporting all stay in the existing pipeline unchanged, and `onAVStarted` (fired by kofin's `Player` after the plugin resolve) is the natural **Ready** trigger the report calls for (§9.4). This keeps SyncPlay decoupled from the play internals — it says *which id at what position*, kofin's normal path decides *how*.
* **Single-process locus: the manager lives in the service.** All group state, timesync, `Api`, and player control are the service's. The root `SyncPlay` item is a transient plugin invocation, so it cannot hold the manager; `mode=syncplay` sends one IPC `SyncPlayMenu` (plugin → service) and the **service** runs `ui.show_menu(manager)` on a **dedicated worker thread** — never the notification thread (the menu blocks on `Dialog().select`/`input`). This mirrors the fork's `SyncPlayMenu`-notification pattern and keeps every piece of SyncPlay state in one process, which the "no cross-process live flags beyond the few legit ones" rule (report §3) wants.
* **WS routing: a SyncPlay branch, manager owns ordering.** `SyncPlayCommand` and `SyncPlayGroupUpdate` arrive on the websocket thread via `Service._on_ws_event`. They route to `manager.on_notification(message_type, data)`, which **enqueues onto the manager's own dispatcher thread** (`Queue` + `syncplay-dispatch`, `manager.py:70-78`) so message ordering is preserved and the WS thread is never blocked. Cleanest seam: extend `RemoteHandler.handle` to claim the two SyncPlay message types (it already owns Play/Playstate/GeneralCommand and returns `True` when handled) *or* add an explicit SyncPlay branch in `_on_ws_event` before the library branch — pick the RemoteHandler route so all "control-plane" WS traffic lives in one place; the handler holds a reference to the manager. `WebSocketConnected`/reconnect (the `ws.py` auto-reconnect already exists) triggers the manager's kicked-probe (below).
* **`onAVStarted` + guarded forwarding (new on kofin's Player).** kofin's `Player` currently has no `onAVStarted` (only `onPlayBackStarted`); phase 4 adds it as the Ready trigger, plus a `_syncplay_event(name, *args)` shim that forwards `onAVStarted`/`onPlayBackPaused`/`onPlayBackResumed`/`onPlayBackSeek`/`onPlayBackStopped`/`onPlayBackEnded` into the manager **only when a manager is attached and in a group**. SyncPlay's own player actions are wrapped in the manager's `programmatic()` context so they don't echo back as user requests (the fork's guard-flag pattern, `manager.py` `_prog_*`; kofin's segment engine already uses an analogous post-seek settle, so the idiom is familiar in this file). The Player gains one attribute — `self.syncplay: Optional[SyncPlayManager] = None`, set by the service at build — alongside the existing `syncplay_group_active` stub.
* **The phase-3 stub is now driven.** `player.py:166` `self.syncplay_group_active` was seeded `False` in phase 3 precisely so the segment engine could withhold Play Next inside a group with no later change to the segment code (phase-3 §7 watch item). The manager sets it `True` while `in_group()` and `False` on leave. Zero edits to `plan_for_crossing`/the checker — the flag already gates `offer_next` (`player.py:589`). Cinema-mode and the near-end Play Next prompt read the same flag (near-end already does, via `offer_next`; cinema, if/when it lands, checks it too — report §9.5.4).
* **Follow-only first (K1), then user-control forwarding + tempo (K2).** Land in the report's order (§9.6): K1 = join/leave + execute commands + report Ready/Buffering, **no** user-control forwarding and **no** tempo (a half-built state still *follows correctly*); K2 = user pause/seek/next in Kodi drive the group, queue mirroring, and the drift controller with `SetTempo`. The risky loop-prone parts (echo suppression under real user input, the tempo control loop) come second on a proven follow-only base. K3 = edge cases (transcode-member dead-zone widening, passthrough seek-ladder, cinema/up-next suppression, the settings group) + strings; K4 = conformance-harness validation + the manual two-device soak.
* **Drift correction via `Player.SetTempo` over JSON-RPC.** `SetTempo` is not in the Python `xbmc.Player` API — call it through `xbmc.executeJSONRPC` (`Player.SetTempo {tempo}`), feature-detected once per session via `Player.GetProperties(["canchangetempo"])` (false under some audio-passthrough configs → **micro-seek ladder** fallback, report §9.2/§9.4). Dead-zone + clamp stay well inside Kodi's 0.75–1.55 envelope; `syncPlayTolerance` sets the dead-zone. This is verbatim from the fork's `playback.py` — the adaptation is only the `executeJSONRPC` call shape (kofin already uses that idiom in `remote.py:_rpc`).
* **Timesync = NTP-over-`/GetUtcTime`, kicked on wake.** min-RTT-of-8 window, 5 greedy pings then 1/30s, `offset_ms = server − local` (fork `timesync.py`). `forceUpdate()` on group join and on **screensaver-deactivate/wake** — the service already has the `GUI.OnScreensaverDeactivated` hook (`main.py:229`, currently only kicks a sync catch-up); phase 4 also forwards it to `manager.on_wake()` (force timesync + re-probe group, report §9.5.6). Stale offsets after sleep are a known failure the report calls out; this closes it.
* **v1-server survival quirks (report §9.5.7).** The manager must survive real 10.11 behavior seen in the report's live traces: a command whose `When` is already in the past → **execute-now and extrapolate** the live position (`playback.py` late-command branch); a `Ready` that bounces with a private Seek → obey it, don't fight it; and **after any WS drop assume you were kicked** → on reconnect, `GET /SyncPlay/List` and auto-rejoin the previous `GroupId` (client-side mitigation of the server's zombie/kick gap, report R2). The reconnect signal is the `ws.py` reconnect + the `on_connected` callback; the manager hooks it.
* **No capability change needed.** SyncPlay eligibility is per active session; the server offers it to any session present on the socket, which kofin already registers (`main.py` `post_capabilities` + the WS session). Nothing added to the CAPABILITIES dict. (If a future probe shows a server gating SyncPlay behind a capability, add it then — YAGNI now.)
* **Strings**: 30550–30599 (SyncPlay settings group + menu/toasts). The fork's `translate()` ids (33262–33295) are fork-local and remap into this range.

## 4. Settings spec

**Deviation from report §4 (recorded):** the report proposed a dedicated **SyncPlay tab**. Phase 4 instead adds a **SyncPlay group to the existing Playback tab** (`category id="playback"`, which already holds group 1 Media Segments and group 2 Play Next from phase 3), as a **new group 3**. Rationale: four settings do not earn a tab, and they sit naturally beside the other player-behavior groups; the tab list stays shorter (Account / Transcoding / Playback / Library / Sync / Interface). The report §4 "Tab: SyncPlay" line is updated in the same commit to read "Playback-tab group."

### SyncPlay group (Playback tab, group 3, 30550+)

| id | type/control | notes |
|---|---|---|
| `syncPlayEnabled` | bool, default **false** | master toggle; gates the root **SyncPlay** entry *and* the manager build — off means no manager thread, no root item (opt-in, since it changes playback behavior) |
| `syncPlayDriftCorrection` | bool, default **true** | enables the `SetTempo` drift controller; off → follow via scheduled commands only, no continuous correction (still tolerant within the dead-zone) |
| `syncPlayTolerance` | int ms, default **75** | drift dead-zone (report §9.4); below this the controller does nothing |
| `syncPlayNotifications` | bool, default **true** | member join/leave and "waiting on X" toasts; off silences the informational notifications (never affects behavior) |

No hidden state settings — group membership is live, not durable (a restart leaves the group, by design; the kicked-probe re-joins only within a session). `syncPlayEnabled` visibility on the root entry is read fresh by the plugin process each root listing.

## 6. Test gates (expands `testing-plan.md` §S4 from its current one-paragraph form)

The scripted second group member is `ref/syncplay-conformance` (`syncplay_kit` — join/command/observe). Assertion surfaces in trust order: Jellyfin `/SyncPlay/List` + the harness's own logged timestamps; Kodi JSON-RPC `Player.GetProperties` (`time`, `speed`, `canchangetempo`); the addon debug log via `kodi-logtail`; screenshots last (pixels prove rendering, never state).

* **S4.1 join + follow (K1)**: join a harness-hosted group; a scheduled Unpause executes within `syncPlayTolerance` of `When` (Kodi `Player` time vs the harness's `When`, offset-corrected); Ready is posted with the correct position; harness-driven Pause/Seek propagate to Kodi (follow-only, before K2 forwarding exists). Never a second player start on a tail-only queue update.
* **S4.2 user-control forwarding (K2)**: pause/seek/next in Kodi drive the group (harness observes the echoed authoritative command); the guard flag prevents an echo loop (no runaway command storm in the log); a Kodi stop offers leave-or-ignore.
* **S4.3 drift correction**: induce drift (harness advances/holds); `SetTempo` correction observed via `Player.GetProperties` `speed` moving off 1.0 within the clamp, then re-converging to the dead-zone; with `canchangetempo=false` (passthrough) the micro-seek ladder holds instead; `syncPlayDriftCorrection=off` tolerates within the dead-zone and never tempo-corrects.
* **S4.4 timesync**: offset stabilizes (min-RTT window); `forceUpdate()` on join produces a fresh offset before the first Ready; a screensaver-deactivate re-syncs (log shows the forced update).
* **S4.5 resilience**: harness drop-out surfaces the group-wait toast; a WS drop → on reconnect the manager probes `/SyncPlay/List` and auto-rejoins the previous group; a late command (`When` in the past) executes-now and extrapolates the live position; sleep/wake leaves and rejoins cleanly.
* **S4.6 interactions**: inside a group the near-end Play Next prompt is withheld (the phase-3 `syncplay_group_active` flag — assert no overlay, no phantom Play Next), cinema/up-next suppressed; selecting an external player hides the SyncPlay root entry.
* **S4.7 manual two-device sanity**: this box + a second client (jellyfin-web or a second Kodi) in one real group — join, play, pause/seek both directions, one drift re-converge, leave — as the final judgment pass.

L1 units port from the fork's `tests/test_syncplay*.py` (~1.8k lines): command scheduling incl. late-`When` extrapolation, the drift-controller branch matrix, the timesync min-RTT window, the queue-mirror tail-diff, and echo-guard suppression. New kofin units: the play-path re-target (a group play resolves through `plugin_url(mode=play)`, not a raw path), the kicked-probe decision, and `syncPlayEnabled` gating the root entry.

## 7. Risks / watch items

* **v1-server behavior under mixed clients is the real unknown, not Kodi.** The report is explicit (§9.6): Kodi's player-control surface is *better* than the browser's; the risk is the 10.11 server's SyncPlay under a mix of web + Kodi members. Mitigation is the phasing itself — K1 ships follow-only, and the v1 survival quirks (§2, report §9.5.7) are handled defensively (execute-now on past `When`, obey private Seeks, assume-kicked on WS drop). The report's Phase-0 *server* fixes (esp. the group-wait deadline) are upstream PRs, not kofin work; kofin must be robust against a server that lacks them.
* **`SetTempo` reachability + audio passthrough.** `Player.SetTempo` is JSON-RPC-only and unavailable under some passthrough configs. Feature-detect via `canchangetempo` every session and fall to the micro-seek ladder; S4.3 must exercise the passthrough path on this box (the seek ladder is correct, just less smooth). Never assume tempo is available.
* **The play-path re-target is the one place the transplant diverges** — it must reproduce the fork's start semantics (correct start index + start position estimate `StartPositionTicks + (serverNow − LastUpdate)` when playing) through kofin's plugin resolve, and pause on `onAVStarted` before posting Ready. Get this wrong and every member starts at a different position. L1 covers the URL/position shaping; S4.1 confirms the live start.
* **Notification-thread blocking.** The menu blocks on dialogs and the WS handler must never block; both are handled (menu on a worker thread, WS messages onto the manager's dispatcher `Queue`) — but the wiring is new, so verify under a reconnect storm that the dispatcher drains and the WS thread stays responsive (part of S4.5).
* **Interaction regressions.** SyncPlay must not perturb the phase-3 player features outside a group: the `syncplay_group_active` flag defaults False and is only ever True while `in_group()`, so segment skipping and Play Next are unchanged when SyncPlay is idle. S4.6 asserts the withhold *inside* a group; a quick S3.3 re-run confirms *no* change outside one.
* **Custom-window budget unchanged.** SyncPlay uses only native `xbmcgui.Dialog` (select/input/ok/notification) — the segment-skip overlay stays the single custom window (report §6). No new skin XML.
* **Screensaver hook double-duty.** `GUI.OnScreensaverDeactivated` now fans out to both the sync catch-up (phase 2) and `manager.on_wake` (phase 4); keep them independent so a broken manager can't suppress the sync kick (degrade, don't die — as `_start_library` already contains its failures).
