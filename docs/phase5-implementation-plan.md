# plugin.video.kofin — Phase 5 implementation plan

Date: 2026-07-19. Operationalizes build phase 5 from `rewrite-research.md` §11 and feature §8.4 — **KofinSyncQueue + addon adoption + full-sync overhaul** (in fork terms: `ref/jellyfin-kodi/docs/sync-improvement-plan.md` §5/§6 Phase 2 *and* Phase 3). Phases 1–4 are complete and live-verified. This is the first phase with **two codebases**: a new sibling repo `jellyfin-plugin-kofinsyncqueue` (C#, clean-room ~1k lines) and the kofin addon changes that consume it. Test gates reference `testing-plan.md` §S5, which this plan expands from its current single-paragraph form into S5.1–S5.9 (doc update in the same commit, per convention). The research §8.4 "decide at implementation" storage choice is decided here (§2) and recorded back into §8.4 in the same commit.

**Phase 5 deliverable**: a KofinSyncQueue server plugin — typed change records with `UpdateReason`/`SeriesId`, query-time Etags, retention cutoff in-band, indexed single-record upserts (killing the official plugin's O(n²) write path, `DbRepo.cs:119-138`) — side-loaded on the server; kofin consuming it as **tier 1** through a change-feed provider abstraction: Etag-matching records dropped **before** download (S2.5's 3067 fetches → 0), additions parent-first and newest-first, image-only updates demoted to artwork-only writes, retention overrun answered by an automatic targeted repair instead of silent loss; and the **full-sync overhaul**: newest-first paging so fresh content is browsable minutes into an initial sync, episodes paged per-library (no per-show request storm), persistent DB connections, watermark honesty across full syncs, and an Update button that prunes by ids+Etag diff instead of re-fetching everything. Tier 2 (official KodiSyncQueue) remains fully supported — the refactor must be provably neutral there before the new plugin is even deployed.

## 1. Scope

**In**: the `jellyfin-plugin-kofinsyncqueue` repo (plugin skeleton, event recorders, LiteDB store, `/Kofin/SyncQueue` + `/Kofin/SyncQueue/Info` endpoints, retention task + config page, xunit suite, jprm-style packaging) and its deployment to the server; the addon's change-feed provider abstraction (`sync/changefeed.py`, strict-typed) with the tier-detection ladder and status line; tier-1 adoption in `sync/library.py` (skip-before-download, ordering, parent prefetch, userdata/watermark from the envelope, image-only demotion + artwork-only application, retention-overrun → targeted repair on **both** tiers, per-class progress counts); the full-sync overhaul in `sync/full_sync.py`/`sync/downloader.py` (DateCreated-desc paging, 3-pass tvshows, persistent connections, watermark-at-start, ids+Etag prune for update mode); the S5.x live gates and performance baselines.

**Out (later phases / deliberately)**: the plugin-free tier-3 fallback (`MinDateLastSaved` catch-up + ID-set reconcile — phase 6; the ladder and record type land now so tier 3 is a third provider, not a redesign); upstream Jellyfin PRs (tombstones, `ItemSortBy.DateLastSaved` — sync-plan §4, not kofin work); Piers live testing (nothing here touches Kodi schemas — L2 fixtures unchanged); the legacy `strm` endpoints (not carried, research §7); per-media mapping DBs / parallel video∥music sync (YAGNI ledger); music newest-first ordering (music full sync keeps its structural artist→album→song order); plugin config surface beyond retention days.

## 2. Decisions locked in phase 5

* **Repo split.** The plugin is a sibling repo `/media/bluecon/dev/jellyfin-plugin-kofinsyncqueue` (own git, own release cadence, clean-room — protocol and defects re-derived from the research/sync-plan analysis, no code copied from the GPL-2 official plugin). **License: GPL-3**, and not only for family unity with the (GPL-3-by-transplant) addon: the Jellyfin assemblies every plugin links — `Jellyfin.Controller`/`Jellyfin.Data` — are published as `GPL-3.0-only` (`MediaBrowser.Controller.csproj:13`), so GPL-2-only à la the official plugin is the *incompatible* choice (an upstream inconsistency, not a convention to copy); clean-room is what makes the choice free. LiteDB is MIT — redistributing `LiteDB.dll` in a GPL-3 artifact is fine. Scaffold mirrors the official repo's packaging shape (`build.yaml`: name "Kofin Sync Queue", fresh GUID generated once at scaffold, `targetAbi: 10.11.0.0`, `framework: net9.0`, artifacts = plugin DLL + `LiteDB.dll`), so the existing sideload/repo tooling applies unchanged.
* **Storage: LiteDB 5, used correctly.** The research left LiteDB-vs-EF open; decided: **LiteDB 5.0.15** — the exact dependency proven in-process on 10.11 by the official plugin, typed POCOs, zero SQL, no coupling to the server's EF migration infrastructure (the "sanctioned long-term direction" buys nothing at this size). The O(n²) was a usage bug, not a LiteDB limit: kofin's store does `EnsureIndex` on `ItemId` (unique) and `LastModified` at open, **single-record `Upsert`** per write, bounded batch lists, and no `FindAll()` ever. Fresh file `kofinsyncqueue.db`, `UserVersion=1`, no upgraders (nothing to migrate).
* **Protocol v1 (the §8.4 shape, pinned).** Two authorized endpoints under `[Route("Kofin")]`, user derived from the access token — no userId in the path, nothing to spoof: `GET /Kofin/SyncQueue?since=<unix-seconds>&types=movies,tvshows,…` and `GET /Kofin/SyncQueue/Info` (§4 for full shapes). `types` is an **include** list (the legacy `filter` exclude-list inversion S2.4 had to confirm dies here; kofin already computes the include set in `fast_sync` and inverts it — tier 1 sends it directly). Unknown type tokens are ignored with a log — never the official plugin's `Enum.TryParse("")` → Movies foot-gun. `since` is required (`400` without it; `0` = everything), compared strictly (`LastModified > since`). `Info` carries `ProtocolVersion: 1`; the addon requires `== 1` and falls to tier 2 on mismatch (adopt-by-probe, never raise a floor — the phase-4 stance).
* **Etag at query time, byte-identical predicate.** The record row stores no Etag; the response computes it per record via `BaseItem.GetEtag(user)` on the item **already loaded** for visibility translation (one `GetItemById` per record, not the official plugin's two) — the same `MD5(DateLastSaved)` string `DtoService` puts in every DTO (report-verified, `BaseItem.cs:2607-2620`, `DtoService.cs:392`), computed at query time so an event storm on one item coalesces into a single current-state comparison. Removed records carry `Etag: null` (no item to load). Failure direction is safe by construction: any divergence means a mismatch, which means a redundant download — never a wrongly-skipped write. A one-item live equivalence check (record Etag vs the DTO Etag kofin stores) is part of step 1's smoke.
* **Event capture + coalescing.** `ILibraryManager.ItemAdded/ItemUpdated/ItemRemoved` (with `ItemChangeEventArgs.UpdateReason` — delivered today and thrown away) and `IUserDataManager.UserDataSaved`, batched ~5 s as the official plugin does (bounded lists). Coalescing on upsert, keyed on ItemId: Added+Updated → **Added** (reasons OR'd, `LastModified` bumped); anything+Removed → **Removed**; Updated+Updated → reasons OR'd. Recorded item kinds are exactly the nine the addon syncs (Movie, BoxSet, Series, Season, Episode, MusicVideo, MusicAlbum, MusicArtist, Audio); `SeriesId`/`SeasonId` captured from episode/season payloads at event time; userdata rows stay full `UserItemDataDto` JSON per user (the good part of the old design). Retention: default **90 days** (not infinite), daily cleanup task, `0` = keep forever ⇒ `RetentionCutoff: 0` in the envelope (overrun impossible).
* **Change-feed provider abstraction (the ladder becomes code).** New `sync/changefeed.py` (strict mypy — new code, not transplant): a `ChangeRecord` dataclass `(id, status, media_type, item_type?, last_modified?, update_reason?, etag?, series_id?, season_id?)`, an envelope `(server_time, retention_cutoff)`, and two providers behind one `detect(api)` — **KofinFeed** (probe `/Kofin/SyncQueue/Info`) then **LegacyFeed** (probe `GetServerDateTime`, adapting the `GetItems` shape into sparse records — optional fields `None`) else tier none. Every enhancement keys off **field presence, not tier identity** (research §8.4), so phase 6's server provider is a third implementation, not a redesign. `library.py` keeps `TIER_*` only for the status line (new `TIER_KOFIN`); the `FastSync` gates (`library.py:438,811`) become `tier != NONE`.
* **Skip-before-download is the same predicate, evaluated earlier.** For records with an Etag, compare `"<etag>|plugin"` against the stored checksum — exactly `check_unchanged`'s test (`fields.py:350,353-357`) — using the existing `get_checksum` query ((jellyfin_id, checksum) rows per media type, `queries_map.py:46`) loaded once per catch-up. Matches are dropped **before chunking**; the post-download short-circuit stays (it still serves tier 2, realtime WS events, and anything that slips through). Userdata never rides this path: it arrives as full DTOs in the same envelope and keeps the phase-2 dedup/`userdata_changed_ids` flow (`library.py:887-898`) unchanged.
* **Ordering + parent prefetch.** Added records sort by (parent rank, newest `LastModified` first) — Series→Season→Episode, MusicArtist→MusicAlbum→Audio, movies/boxsets/musicvideos flat — then chunk (`DOWNLOAD_CHUNK`), so the first pages fetched are the newest content; with 3 download workers minor interleaving is accepted (ordering is priority, not serialization). Episode/season records whose `series_id` is unknown to kofin.db get the parent **prepended** as its own addition (dedup'd) — the R8 orphan re-queue dance stays as fallback but never engages on tier 1. The progress label reports per-class counts ("New: 12 | Updates: 340", sync-plan §3) so a metadata backlog is visibly not blocking new content.
* **Image-only demotion → artwork-only write.** Records whose reason is **exactly** `ImageUpdate` route to the lowest priority (behind metadata), fetch with minimal fields (`Fields=Etag` — image tags are DTO defaults), and apply through a new contained `artwork_only` path: resolve (kodi_id, media) from kofin.db, `Artwork.add(get_all_artwork(map(item, "Artwork")), …)` (the seam every writer already uses, e.g. `writers/movies.py:142`), stamp the checksum. Video types only (Movie/Series/Season/Episode/MusicVideo); boxsets and music take the full path; **any** anomaly (no mapping row, unexpected payload) falls back to the full update queue. This is writer-*adjacent*, not a writer change — the cascade itself is untouched, and an L2 fixture proves art-table equivalence with a full write.
* **Retention overrun: detected in-band, answered automatically — on both tiers.** Tier 1: envelope `RetentionCutoff > 0` and `since < cutoff` ⇒ overrun. Tier 2: read `RetentionDateTime` from the `GetServerDateTime` response — returned today and ignored by every client (`save_last_sync` uses only `ServerDateTime`, `library.py:938`). Response: one notification + enqueue the **update-mode pass** (catch-up + prune, below) for all whitelisted libraries; the watermark advances only after it completes. Converts R5's silent loss into a self-healing event; completes S2.10's retention half.
* **Watermark rules tightened.** Tier 1 advances to the **feed envelope's `ServerTime`** — same response, zero extra round trips (speed item 7), no clock skew, so the tier-2 `−2 min` fudge and its `GetServerDateTime` round trip (`library.py:927-952`) apply only on tier 2. Full syncs: stamp `lastIncrementalSync` with server time **at the start** of the very first full sync (empty watermark), and **never advance it at full-sync end** — delete the `save_last_sync()` call at `full_sync.py:188`. That call is a correctness hole: `add_library` for library B jumps the watermark past pending queue records for libraries A/C (lost until repair), and the initial-sync window survives only via the fudge. With watermark-at-start the first catch-up replays everything that changed during the sync — Etag skip makes the replay nearly free — and the incremental path (`library.py:376`, drain-success only) becomes the watermark's sole owner. WS events during post-startup syncs already buffer by construction (they enqueue from the notification thread, `main.py:284-291`, and drain when the library loop resumes); events during *startup* stay gated (`main.py:281`) and are now covered by the watermark instead of lost — the sync-plan Phase-3 "buffer and replay" requirement, satisfied without a buffer.
* **Full-sync overhaul.** (a) Video paging flips to `SortBy=DateCreated,SortName` / `SortOrder=Descending,Ascending` (`downloader.py:124,149`; the SortName tiebreak keeps pagination deterministic under equal timestamps; the 10.11 composite `DateCreated` indexes make it cheap, sync-plan §4.4) — fresh content is written first and browsable minutes into an initial sync; music keeps `AlbumArtist` order (structural). RestorePoints already record full query params, so an in-flight pre-flip restore point resumes under its recorded sort — no migration. (b) **tvshows becomes three per-library passes** — Series pages, Season pages (`/Items?IncludeItemTypes=Season&Recursive`), Episode pages — replacing the per-show `get_episode_by_show` loop (`full_sync.py:407`; 76 shows ⇒ 76 requests today, P5's 500-show pattern): restore keys `{lib}/tvshows-{series,seasons,episodes}`; a pre-phase-5 pending `{lib}/tvshows` key restarts that library's passes (idempotent writes make it safe); the writer's season-on-demand fallback stays for any season the Items query misses. (c) **Persistent connections**: `Database()` handles held across a library's pass, the writer **lock still taken and the transaction committed per page** — holding the lock across a pass would starve realtime writers, so the win is scoped to connection churn; this narrows sync-plan Phase-3's "commit every N items" wording, recorded here as the deviation. (d) **Update mode becomes the ids+Etag prune** (research §3 "update that works"): per library per type, page `Id`+`Etag` only (`Fields=Etag&EnableUserData=false`, limit 500 — a new light pager beside `_get_items`), diff against the `get_checksum` map three ways — missing locally → fetch (added queue); stale locally → remove via writers; Etag mismatch → fetch (updated queue); match → nothing. S2.5's Update (full re-fetch of 3067 records, 0:41) becomes ~7 id-pages + zero fetches on a clean tree. Repair (drop & full resync) is untouched.
* **No new settings.** Zero user-facing surface: the Sync tab status line gains the tier-1 string, and research §4's "kodiCompanion auto-detected" design is thereby complete. **Strings: 30600–30619** (tier-1 status line, retention-overrun notification, targeted-repair toast).

## 4. Protocol spec (pinned for v1)

```
GET /Kofin/SyncQueue/Info                          [Authorize]
→ { "PluginVersion": "1.0.0.0", "ProtocolVersion": 1,
    "ServerTime": <unix>, "RetentionCutoff": <unix|0>, "RetentionDays": 90 }

GET /Kofin/SyncQueue?since=<unix>&types=movies,tvshows,boxsets,musicvideos,music   [Authorize]
→ { "ServerTime": <unix>, "RetentionCutoff": <unix|0>,
    "Items": [ { "Id": "<guid>", "Status": "Added|Updated|Removed",
                 "MediaType": "movies|tvshows|boxsets|musicvideos|music",
                 "ItemType": "Movie|BoxSet|Series|Season|Episode|MusicVideo|MusicAlbum|MusicArtist|Audio",
                 "LastModified": <unix>, "UpdateReason": "ImageUpdate, MetadataEdit, …",
                 "Etag": "<md5>|null, "SeriesId": "<guid>|null", "SeasonId": "<guid>|null" } ],
    "UserData": [ <full UserItemDataDto>, … ] }
```

Semantics: records are the caller's user-visibility-translated view; `types` filters by include; `Items` is coalesced (one record per id, §2); `UserData` is the caller's rows only. The addon's canonical watermark stays the ISO `lastIncrementalSync` setting — providers convert (unix for Kofin, `LastUpdateDT` string for legacy) so tier switches mid-life never strand it.

## 7. Risks / watch items

* **Touching the proven full-sync path is the riskiest part of the phase** — the writers stay untouched (paging changes only order of arrival), but the pager/restore-point/prune changes sit in transplanted code. Mitigation: the S5.8/S5.9 gates re-prove exactly what S2.1/S2.2/S2.7/S2.10 proved (counts, fidelity, resume, heal), and step 4 lands separately from the tier-1 work so a regression bisects cleanly.
* **The Etag-equivalence assumption is load-bearing for the tier-1 win** but fails safe (mismatch = redundant download, never a skipped write). The step-1 smoke checks it against the live server; if 10.12+ ever changes Etag derivation the symptom is "tier 1 stops skipping," caught by the S5.3 request-count baseline on re-runs.
* **Coalescing correctness under event storms**: Added-then-Removed inside one batch must emit Removed (the addon tolerates removes for unknown ids — the SortWorker no-ops them); the 5 s batcher must stay bounded during a library scan. The xunit matrix + the S5.7 storm cover both.
* **Realtime starvation during full syncs**: the persistent-connection change deliberately keeps lock+commit per page (§2) — verify during S5.8 that a realtime userdata event lands between pages, not after the library completes.
* **Remote deploy loop**: plugin iteration needs a server restart per install; batch plugin-side changes and lean on xunit locally so live deploys are rare. The official plugin stays installed throughout — stock jellyfin-kodi clients on the server keep working, and the tier ladder is exercised in its real shape.
* **Tier-2 neutrality is a hard gate, not a hope**: step 2's S2.3/S2.4/S2.5 rerun happens *before* tier-1 code exists; any drift there is a refactor bug by definition.
* **Masking**: the new endpoints carry no token in the URL and no userId in the path by design; Etags are not secrets. The grep chokepoint check still runs over a full tier-1 session (step 5) because that is the rule, not because a leak is expected.
